\begin{question}{Basic deployment pipeline -- Modeling with Petri Nets}{3.5}
	Incoming update requests are submitted to the deployment service. Each request
	corresponds to a deployment batch, which must pass through a sequence of
	processing stages before being installed on devices. \\

	The pipeline includes the following shared resources:
	\begin{itemize}
		\item Validation Service -- verifies update packages before deployment.
		\item Deployment Manager -- coordinates sending updates to devices.
		\item Two Deployment Workers -- workers that push updates to devices in parallel.
		\item Staging buffer -- can hold at most three deployment batches simultaneously. \\
	\end{itemize}

	The validation service, the deployment manager, and the deployment workers
	process only one batch at a time. \\

	Each deployment batch goes through the following stages:
	\begin{itemize}
		\item Submission -- an update request enters the system.
		\item Staging -- the batch waits in the staging buffer.
		\item Validation -- the validation service checks the update package.
		\item Dispatch -- the deployment manager assigns the batch to an available worker.
		\item Deployment -- a worker performs the update rollout.
		\item Completion -- the batch finishes deployment successfully. \\
	\end{itemize}

	Model the system in a Petri net.
\end{question}

\begin{solution}
	My Petri net model for the basic deployment pipeline is shown in the figure below. I will briefly describe the key design decisions.

	\begin{itemize}
		\item The Update\_Requests place has an initial marking of 1 with a self-loop on the Submit\_Request transition, allowing an unbounded number of update requests to enter the system over time.
		\item The staging buffer capacity of 3 is enforced by an Available\_Space\_In\_Buffer resource place initialized to 3. The Stage\_Deployment\_Batch transition consumes one token from this place, and the Validate\_Deployment\_Batch transition returns it, ensuring at most 3 batches occupy the buffer at any time.
		\item The Validation\_Service place (initialized to 1) enforces mutual exclusion on validation. It is consumed by Validate\_Deployment\_Batch and returned by either Deployment\_Batch\_Is\_Valid or Deployment\_Batch\_Is\_Invalid, ensuring only one batch is validated at a time.
		\item The Deployment\_Manager place (initialized to 1) enforces mutual exclusion on dispatch. It is consumed and returned by the Assign\_To\_Worker transitions.
		\item Worker\_1 and Worker\_2 (each initialized to 1) model the two deployment workers. When a batch is assigned to a worker, the worker token moves to a Worker\_Busy place and is returned upon rollout completion.
		\item After validation, a nondeterministic choice between Deployment\_Batch\_Is\_Valid and Deployment\_Batch\_Is\_Invalid models the possibility of invalid batches.
	\end{itemize}

		\caption{Petri net model for the basic deployment pipeline.}
		\label{fig:q1-net}

\end{solution}

\begin{subquestion}
	Resource exclusivity -- The validation service and deployment manager are
	never used by more than one batch at the same time.
\end{subquestion}

\begin{solution}

	\begin{step}
		CTL Expression
	\end{step}

	To verify this, I check that the resource tokens for both the validation service and the deployment manager are conserved in a way that prevents concurrent use. Specifically, I verify two things simultaneously:

	\begin{itemize}
		\item The Deployment\_Manager place always contains exactly 1 token (it is never consumed without being returned).
		\item The sum of tokens in Validation\_Service and Validated\_Deployment\_Batches is always exactly 1. This captures the invariant that at any given time, the validation service is either idle (token in Validation\_Service) or processing exactly one batch (token in Validated\_Deployment\_Batches), but never both and never neither.
	\end{itemize}

	The CTL expression is:

	\begin{align*}
		AG\Big(&\text{TAPN1.Deployment\_Manager} = 1 \\
		&\;\land\; \text{TAPN1.Validation\_Service} + \text{TAPN1.Validated\_Deployment\_Batches} = 1\Big) \\
	\end{align*}

\begin{subquestion}
	Pipeline progress -- If a batch enters the system, is it always possible for it to
	eventually complete deployment?
\end{subquestion}

\begin{solution}

	\begin{step}
		CTL Expression
	\end{step}

	This is a liveness property asking whether, from any state in which a batch has been submitted, it is possible (not necessarily guaranteed) for the batch to eventually reach completion. The ideal CTL formulation is:

	\begin{align*}
		AG\left(\text{TAPN1.Submitted\_Update\_Requests} = 0 \;\lor\; EF\left(\text{TAPN1.Completed\_Updates} \ge 1\right)\right) \\
	\end{align*}

	This encodes the implication: ``in every reachable state, if a batch has been submitted, then there exists a path from that state along which some batch eventually completes.'' The implication $P \to Q$ is encoded as $\lnot P \lor Q$, where $P$ is ``a batch has been submitted'' and $Q$ is ``completion is eventually reachable.'' \\

	However, this formula contains nested path quantifiers ($AG$ wrapping $EF$), which can be computationally expensive for the model checker. As a practical alternative, we can verify the non-nested formula:

	\begin{align*}
		EF\left(\text{TAPN1.Completed\_Updates} \ge 1\right) \\
	\end{align*}

	This simply asks: ``is it possible for a batch to ever reach the Completed\_Updates place?''

\begin{solution}
	The hardest part of this assignment was, without question, formulating and verifying the CTL queries for Questions 2 and 4. While the Petri net modelling itself was fairly mechanical once I understood the system requirements, the CTL formulation process exposed several challenges that I did not anticipate going in. \\

	The first major difficulty was the ambiguity of the natural language descriptions in the assignment. Several of the requirements could be reasonably interpreted in more than one way. For instance, ``critical updates that enter the system eventually complete deployment'' could mean that completion is guaranteed on all paths ($AF$), or that completion is always at least possible ($EF$). The choice between these two readings changes not only the formula but also what the result actually tells you about the system. I found myself spending more time reasoning about what the question was really asking than I did constructing the formula itself. \\

	The second challenge was computational. Several of the CTL formulas I came up with were correct in principle but unverifiable in practice. Specifically, any formula with nested path quantifiers (such as $AG$ wrapping $AF$, or $AG$ wrapping $EF$) caused TAPAAL's verifier to either run indefinitely or nearly crash my machine. This was especially frustrating for Question 4b, where the ideal formula for critical update progress required exactly this kind of nesting. I had to resort to weaker, non-nested alternatives and supplement them with structural reasoning across multiple query results to build a convincing argument. This taught me an important lesson: the gap between what you can express in a logic and what you can actually verify with a tool is significant, and working within a tool's limitations is itself a skill. \\

	The third difficulty was the limitation of working with uncoloured Petri nets. Since all tokens in an uncoloured net are anonymous, there is no way to track a specific deployment batch from submission through to completion. This meant that even when I could verify that some batch eventually completes, I could not verify that a particular critical batch does. The Completed\_Updates place accumulates tokens from both regular and critical pipelines indistinguishably. A coloured Petri net would have allowed me to tag tokens by type and trace them individually, but that was outside the scope of this assignment. \\

	On the broader questions posed by this module: I think the most important takeaway is the distinction between verification and validation. Verification asks whether a model satisfies a specification with mathematical certainty, while validation asks whether the model itself faithfully represents the real system. In this assignment, I could verify properties of my Petri net, but whether my net correctly captures the deployment pipeline described in the problem is a separate question entirely. Both are necessary, and neither is sufficient on its own. \\

	In a professional setting, I expect I would rely primarily on validation (testing, code review, integration checks) for day-to-day work, simply because formal verification is expensive and the tooling is often impractical for large systems. That said, for safety-critical components where correctness is non-negotiable, formal methods would be well worth the investment. This assignment demonstrated that clearly: the properties I could verify gave me genuine confidence in my model's correctness, far beyond what manual inspection alone would provide. \\

	Finally, on the role of tooling: TAPAAL made this entire exercise feasible. Without it, verifying even simple CTL properties on a net of this size would require exhaustive manual state space exploration, which is obviously impractical. At the same time, the tool's limitations (bounded verification, inability to handle nested quantifiers efficiently) were themselves a source of frustration. Good tooling does not just make formal methods accessible, it largely determines whether they get adopted at all.
\end{solution}
